# AGENTS 开发协作规范（FinFraud-RAG）

本文档用于统一本项目的开发流程、目录边界、代码风格、测试策略与交付要求。所有后续开发均应遵循本文档。

## 1. 项目目标与范围

本项目目标是构建一个可持续积累诈骗知识的金融风险治理系统，包含：
- 离线知识库构建：将诈骗文本经 LLM 分析后结构化入库并向量化。
- 在线检索与判别：关键词检索 + 向量检索双通道召回，结合 LLM 完成风险判断。
- 对比实验：在给定数据集上完成「有 RAG vs 无 RAG」准确率评估并输出完整日志。

## 2. 强约束目录规范

必须保持前后端分离，且目录职责清晰。

- `backend/`
  - 后端主目录（Python + FastAPI + Chroma）。
  - `backend/service/` 仅存放接口层代码（API 路由、请求响应模型、依赖注入等）。
  - 除 `service/` 外的后端处理逻辑必须放在独立模块（如 `core/`、`rag/`、`storage/`、`eval/` 等），禁止把业务流程代码塞进 `service/`。
- `webui/`
  - 前端目录（Next.js），仅包含界面与前端交互逻辑。
- `data/`
  - 数据集目录，作为输入数据来源，不在开发流程中破坏原始文件。
- `SPECS/`
  - 阶段规划与验收标准文档。
- `README.md`
  - 项目统一使用说明、阶段测试方式说明、依赖说明、实验说明。

## 3. 技术栈与工具链约束

- 后端：Python（推荐 3.11+）
- API：FastAPI
- 前端：Next.js
- 向量数据库：Chroma
- 包管理：`uv`

规则：
1. 新增 Python 依赖后，必须更新 `pyproject.toml`（及锁文件）与 `README.md`。
2. 所有运行命令、测试命令使用 `uv` 体系表达。
3. 保持可复现：固定关键依赖版本范围，避免漂移。

## 4. 功能实现原则

### 4.1 离线知识库构建

单条诈骗文本处理链路必须包含：
1. 一次 LLM 调用完成结构化分析，输出至少包含：
   - `summary`：完整句子形式的诈骗关键信息摘要（用于向量匹配）
   - `category`：诈骗类型（投资/贷款/招聘/奖金等，可扩展）
   - `pattern`：诈骗套路总结（可多条）
   - `risk_keywords`：风险特征关键词（可多条）
2. 将原文本 + LLM 结构化结果写入存储。
3. 对 `summary` 做 embedding，并写入 Chroma 用于相似度召回。

### 4.2 在线检索与判别

对用户输入文本执行：
1. 关键词提取并做关键词匹配召回（top-k）。
2. 文本向量化并对摘要向量做相似度召回（top-k）。
3. 融合两路召回结果（去重、排序、截断、保留证据）。
4. 将融合结果作为上下文喂给 LLM，输出诈骗判别结论及理由。

### 4.3 实验与评估

必须实现自动化评估程序：
1. 从 `data/job_scams/train*.jsonl` 与 `data/sms/train*.jsonl` 抽取 `label=1` 文本用于离线构建。
2. 从两个数据集 `test*.jsonl` 各取前 500 条组成测试集。
3. 分别评估：
   - 无 RAG：仅依赖 LLM 判别。
   - 有 RAG：使用双检索 + 上下文增强判别。
4. 输出完整实验日志，至少包含：
   - 输入文本
   - 真实标签
   - 模型回答
   - 最终预测
   - 是否命中
   - （RAG）召回证据摘要
5. 输出整体准确率及必要分项指标（建议：precision/recall/F1）。

## 5. 编码与接口规范

- 统一使用类型标注与 Pydantic 模型。
- API 输入输出必须有明确 schema；错误码与错误信息可追踪。
- 配置集中管理（如 `.env` + settings 模块），禁止硬编码密钥。
- 关键流程必须可观测：结构化日志、阶段耗时、错误堆栈。
- 模块边界清晰：
  - `service/`：仅协议层与编排入口。
  - 业务逻辑：独立到 `core/` 或同级模块。
  - 存储逻辑：独立到 `storage/`。
  - LLM/Embedding 适配器：独立到 `llm/` 或 `providers/`。

## 6. 测试与质量门禁

每一阶段新增功能后必须同步提供：
1. 如何运行该功能（命令、参数、示例输入）。
2. 如何验证结果正确性（预期输出、检查点）。
3. 将以上内容更新进 `README.md`（阶段性）与最终统一说明（全局）。

建议测试层级：
- 单元测试：解析、融合、排序、数据清洗等纯函数。
- 集成测试：离线构建流水线、在线检索接口。
- 端到端冒烟：前端提交文本 -> 后端返回判别。
- 实验评测：RAG 对比基线准确率。

## 7. README 更新约定

每完成一阶段功能必须更新 README，至少包含：
- 新增依赖与安装方式（`uv`）
- 启动命令与环境变量
- 新接口说明与示例
- 新增测试方法
- 已知限制

项目全部完成后，README 必须提供统一入口：
- 一键或最短路径启动后端/前端
- 一键或最短路径执行离线构建
- 一键或最短路径执行对比评估
- 输出产物位置说明（日志、指标、向量库目录）

## 8. 数据与安全合规

- 不泄露敏感配置（API Key、私密路径）到代码仓库。
- 日志中如含敏感字段需脱敏（手机号、身份证、银行卡等）。
- 对外返回结论需包含“不构成法律或金融建议”类提示（如适用）。

## 9. 里程碑交付要求

每阶段交付应满足：
1. 代码可运行。
2. README 已更新可复现步骤。
3. 对应 SPECS 验收项可逐条核对。
4. 不破坏整体架构一致性。

## 10. 变更管理

- 涉及目录职责变更、核心数据结构变更、检索融合策略大改时，先更新 `SPECS/` 对应文档再实施。
- 实施后同步更新 README 与必要注释，保证后续维护者可追溯。
