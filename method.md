# 方法说明（method.md）

## 1. 方法总览

本项目采用“离线知识构建 + 在线检索增强判别”的方法体系，核心目标是：

1. 将诈骗文本转化为结构化知识并持续沉淀到向量库；
2. 对用户输入执行双通道检索（BM25 + 向量），再将证据拼接进模型上下文；
3. 利用上下文增强模型对输入文本进行诈骗风险判断。

从方法角度可分为两大流程：
- 离线处理入库方法
- 在线检索与上下文拼接方法

---

## 2. 离线处理入库方法

离线入库由统一的入库流水线执行，核心步骤如下。

### 2.1 输入与标准化

输入单元为文本记录（`text` + `source`）。
- `text`：待分析文本
- `source`：来源标识（如 `manual`、`job_scams:train`）

离线脚本或接口会先对输入进行基础合法性约束（非空、结构完整）。

### 2.2 LLM 结构化抽取（一次调用）

对每条文本进行一次 LLM 调用，抽取结构化字段：
- `summary`：诈骗关键信息摘要（后续用于向量召回）
- `category`：诈骗类别
- `patterns`：诈骗套路
- `risk_keywords`：风险关键词

输出要求为 JSON，以确保可解析和可落库。

### 2.3 入库前过滤规则（关键）

当前项目在入库前执行强约束过滤：

- 若 `risk_keywords` 为空列表，则该条文本**不入库**；
- 返回跳过信息（如：未提取到诈骗信息）；
- 在日志中标记为 `skipped`。

该规则在所有入库路径统一生效：
- 前端上传/单条入库
- CLI 离线入库
- 实验阶段 train 构建入库

其方法意义是降低噪声知识进入向量库，提升检索质量与判别稳定性。

### 2.4 摘要向量化

对 `summary` 调用 Embedding 服务生成向量。
- 向量化目标字段固定为 `summary`
- `patterns/risk_keywords` 作为结构化 metadata 保存

### 2.5 向量库存储（Chroma）

每条通过过滤的记录写入 Chroma：
- `document`：`summary`
- `embedding`：摘要向量
- `metadata`：`category/patterns/risk_keywords/original_text/source/created_at`

### 2.6 结果回传与日志

入库结果分三类统计：
- `success`：成功写库
- `failed`：执行失败
- `skipped`：命中过滤规则未写库

并写入结构化日志（实时追加），保证中断场景下结果可追溯。

## 3. 在线检索与上下文拼接方法

在线流程在用户提交文本后执行，目标是构造高质量证据上下文供模型判别。

### 3.1 查询预处理

对用户输入执行：
- 文本标准化（空白压缩、去首尾空格）
- 分词/切词（用于 BM25 query token）

### 3.2 通道A：BM25 词法检索

#### 3.2.1 文档构造

每条知识文档并非仅使用 `summary`，而是拼接三字段：

`summary + patterns + risk_keywords`

这样可以同时覆盖：
- 语义摘要
- 套路短语
- 显式风险词

#### 3.2.2 打分

项目使用自实现 BM25（非 Chroma 原生）。
- 对 query token 与语料 token 建索引
- 计算每条文档 BM25 分数
- 归一化后取 top-k

该通道强化“关键词/术语显式命中能力”。

### 3.3 通道B：向量相似度检索

对用户输入生成 query embedding，在 Chroma 上检索 `summary` 向量：
- 返回 top-k 候选
- 根据距离映射相似分数（0~1）

该通道强化“语义近邻召回能力”。

### 3.4 双通道融合

将 BM25 与向量结果按 `record_id` 去重合并，采用加权融合：

`fused_score = 0.4 * bm25_score + 0.6 * vector_score`

并按融合分数排序，形成最终证据序列 `fused_hits`。

### 3.5 证据压缩与上下文拼接

为减少 token 消耗，证据上下文不直接放长 UUID：
- 将排序后的证据重编号为 `ref_id=1,2,3...`
- 上下文中使用 `ref_id`、分数、类别、摘要等关键信息

示意：
- `ref_id=1 | score=... | category=... | summary=...`
- `ref_id=2 | ...`

这种方法可显著压缩上下文长度，提升上下文利用效率。

### 3.6 模型判别与证据映射回填

模型基于“用户文本 + 拼接证据”输出结构化结论：
- `is_scam`
- `confidence`
- `reason`
- `evidence_refs`

其中 `evidence_refs` 若返回短编号（如 `1,2`），系统会再映射回真实 `record_id`，保证外部结果可追踪。

## 4. 评估方法（与主流程关系）

项目评估方法复用了上述完整链路：

1. 用 train 正样本做离线构建（含 `risk_keywords` 过滤）
2. 在 test 前 N 条样本上执行判别
3. 输出逐条日志与汇总指标（accuracy/precision/recall/F1）

支持三种评估模式：
- NoRAG（仅 LLM）
- RAG（双检索 + 上下文）
- RAG-only（只评估 RAG）

并采用实验隔离 collection，避免污染线上默认知识库。

---

## 5. 方法特点与价值

从方法层面，本项目的关键特点是：

1. 结构化抽取驱动知识沉淀：将非结构化文本转化为可检索知识单元；
2. 入库质量门控：`risk_keywords` 为空即跳过，减少噪声；
3. 双通道互补召回：BM25 负责词法命中，向量负责语义近邻；
4. 融合排序与证据压缩：提升判别上下文质量并控制 token 成本；
5. 可复现实验体系：同一方法链路可用于线上推理与离线评估。

整体上，该方法使系统具备“可持续积累 + 可检索增强 + 可评估迭代”的闭环能力。
